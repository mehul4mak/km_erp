What Odoo actually is

Odoo is an open-source ERP — a big bundle of business software (Sales, Purchase, Manufacturing, Inventory, Accounting…) that all shares one database. We run the free Community edition in Docker, next to a PostgreSQL database.

Under all the apps, Odoo is really three things stacked up:

A data model

Hundreds of predefined business tables — customers, products, orders, bills of material, stock moves. Odoo calls each one a model (e.g. sale.order).

An ORM + business logic

Every model is a Python class with methods. Some are plumbing (create, write); some are real operations (action_confirm) that run Odoo's engine.

A web layer

Odoo's own UI, plus an External API that exposes every model and method over HTTP. That API is our doorway.

Postgres underneath

All data lives in one PostgreSQL database. We almost never touch SQL directly — the ORM does it for us.

The big idea

We don't rebuild manufacturing software. Odoo already knows how a factory works. Our frontend just drives Odoo from the outside over its API, and hides Odoo completely behind a branded UI. The customer never knows Odoo is there.

The core

The one API call you need to understand

Odoo's External API is almost comically simple: everything is one endpoint. You POST JSON that says "on this model, run this method, with these arguments." Odoo runs the Python method and returns the result. That endpoint is /web/dataset/call_kw.

Before you can call it, you log in once to get a session cookie:

lib/odoo.js — authenticate once, reuse the cookie
POST /web/session/authenticate
{ "db": "erp", "login": "svc_portal", "password": "••••" }
// → returns a session cookie we cache and attach to every later call

After that, one function — call_kw — is 90% of the whole integration. Our wrapper in lib/odoo.js looks like this:

The shape of every single Odoo call we make
callKw(model, method, args, kwargs)
   //  model  = which Odoo table, e.g. "sale.order"
   //  method = which function, e.g. "action_confirm"
   //  args   = positional arguments (usually starts with record IDs)
   //  kwargs = named options, e.g. { limit: 8, order: "date_order desc" }

The most common method is search_read — "find records matching a filter and give me these fields." We wrap it as searchRead(model, domain, fields, kwargs). A domain is Odoo's filter language: a list of [field, operator, value] conditions.

A real read from the dashboard (app/page.js)
searchRead(
  "mrp.production",                              // model: manufacturing orders
  [["state", "not in", ["done", "cancel"]]],  // domain: still open
  ["name", "product_id", "product_qty", "state"], // fields we want back
  { limit: 8 }
)
Why it matters

Because every read and every action is the same call, the entire backend integration lives in two small files: lib/odoo.js (the transport) and lib/actions.js (the writes). That is the thin boundary that keeps Odoo swappable later.

The verbs

Two kinds of methods: boring and powerful

Every method we call falls into one of two buckets. The distinction is the single most important idea in this whole document.

Plumbing / CRUD

Move data in and out

create, read, write, search_read. These just store and fetch records. Nothing clever happens.

Business methods

Run the factory

action_confirm, button_mark_done, button_confirm. These trigger Odoo's engine — one call can create many records and enforce rules.

Here is the complete list of what MixerWorks actually calls, pulled straight from lib/actions.js:

Notice

Three of those business methods are Odoo's own (action_confirm, button_confirm, button_mark_done) and two are ours. To the API they're identical — our custom methods are called exactly the same way as Odoo's built-ins. That's the power of building on Odoo rather than beside it.

The payoff

One click, traced end-to-end

This is where Odoo earns its keep. When a user creates and confirms an order for 10 × REMI STORM, our code makes just two API calls. Odoo does the rest — automatically.

lib/actions.js — this is the entire code
// 1. create the quotation
const soId = await callKw("sale.order", "create", [{
  partner_id: 7,
  order_line: [[0, 0, { product_id: 42, product_uom_qty: 10 }]],
}]);

// 2. confirm it — and stand back
await callKw("sale.order", "action_confirm", [[soId]]);

That second line sets off a chain reaction inside Odoo, driven by the product's routes (Make-To-Order + Manufacture/Buy):

1
Sales order confirmed
sale.order → state: "sale"

The customer demand for 10 mixers is now real. Odoo asks: how do we fulfil it?

2
Manufacturing Order created — 10 × REMI STORM
mrp.production (auto)

The mixer is "manufactured", so Odoo raises a factory job and reads its Bill of Materials to see what it needs.

3
Sub-assembly MO created — the motors
mrp.production (nested)

The BOM says each mixer needs a Motor 35mm, which is also manufactured. So Odoo raises a second, nested job to build the motors first.

4
Purchase order raised — ~33 component lines
purchase.order (auto, to the vendor)

Every part that's bought, not made, gets gathered into an RFQ to the supplier — at the prices from the vendor pricelist.

✓
Our pages read it all back
search_read on each model

The Orders page shows the linked jobs and POs; Production shows live material readiness; Inventory shows what's now short. No extra code — just reads.

The lesson

Building that cascade by hand — BOM explosion, nested sub-assemblies, procurement, vendor pricing, stock reservation — would be months of careful, bug-prone work. We got it from one action_confirm call. That is why we chose Odoo.

Things that surprise newcomers

Three Odoo quirks worth knowing early

1 · The [0, 0, {…}] command syntax

When you write a one-to-many field (like adding lines to an order), Odoo doesn't take a plain list. It takes command tuples. [0, 0, {values}] means "create a new linked record." That's why our order creation looks the way it does:

order_line: [[0, 0, { product_id: 42, product_uom_qty: 10 }]]
//         └ 0 = "create"
// other commands you'll meet: [6,0,[ids]] = "replace all links with these"
//                             [4, id]     = "add this link"

2 · A link comes back as [id, "name"]

Read a Many2one field (like a partner) and Odoo returns a pair, not just an ID. That's why the UI code reaches for the second element:

// so.partner_id is  [7, "Sharma Distributors"]
so.partner_id?.[1]   // → "Sharma Distributors" (the label to show)
so.partner_id?.[0]   // → 7 (the ID to filter on)

3 · Template vs variant

A product.template is the catalogue item ("REMI STORM"). A product.product is the concrete sellable/stockable variant of it. Stock quantities and BOM lines point at the product.product. It trips everyone up once; then it never does again.

The business shape

Why this is a SaaS — not just an Odoo install

SaaS — Software as a Service — means customers rent software that runs on your servers and pay a subscription, instead of buying and installing it themselves. Three things turn our Odoo setup into a SaaS product:

A
White-label — Odoo is invisible

The customer only ever loads the branded Next.js app. Every Odoo call happens server-side as svc_portal; no Odoo URL, cookie, or screen ever reaches the browser. To the customer it's bespoke software called MixerWorks.

B
You host and operate it

Odoo, Postgres, and the frontend run on your infrastructure. Customers need nothing but a browser. You handle updates, backups, and uptime — that's the "service" part.

C
One product, many customers

The same codebase serves every manufacturer you sign. The open question — Odoo's natural model is one database per customer — is the main thing to design before customer #2. See docs/07-ROADMAP.md.

The mental model

Think of Odoo as a powerful engine you bought, and MixerWorks as the car you built around it. Customers drive the car. They never open the hood — and if you ever swap the engine, they won't notice, as long as the dashboard keeps working.

Reference

Glossary

ERP
Enterprise Resource Planning — one system running sales, purchasing, manufacturing, and stock together. Odoo is one.
SaaS
Software as a Service — customers rent hosted software on subscription instead of installing it.
ORM
Object-Relational Mapping — the layer that lets you work with database rows as objects/methods instead of writing SQL.
Model
Odoo's word for a business table+class, e.g. sale.order.
JSON-RPC
A simple "call a remote function over HTTP with JSON" protocol. Odoo's External API speaks it.
call_kw
The single endpoint that runs any model method remotely. Our whole integration rides on it.
Domain
Odoo's filter format: a list of [field, operator, value] conditions.
BOM
Bill of Materials — the recipe of components (and quantities) for a product.
MO
Manufacturing Order — one production job (mrp.production).
MTO
Make-To-Order — a route where confirming a sale automatically triggers production/purchase.
white-label
Selling someone else's engine under your own brand. Community's LGPL licence permits it.
svc_portal
The single Odoo service account the frontend logs in as to make every API call.
