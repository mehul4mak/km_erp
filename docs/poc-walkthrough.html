1

Dashboard — the factory at a glance

The landing screen: confirmed order value, quotes in the pipeline, jobs on the floor, purchases pending, and components to replenish — all computed live from Odoo. Recent orders and current production jobs sit side by side. Notice the branding: this reads as bespoke manufacturing software called MixerWorks.

2

Cost Sheet — the Excel replacement

Pain · stale cost sheets

This is the live cost sheet for the in-house Motor 35mm. Its ten parts roll up to ₹561.00 of material, labour is added, then rejection % and profit % build to a suggested price — recomputed instantly when any rate changes. No more spreadsheets going stale. Edit a percentage, hit Recalculate, apply the price to quotations.

3

Order — one quote, a whole cascade

Connected flow

The confirmed order for 10 × REMI STORM at ₹1,732.51 — the exact price the cost sheet produces. A progress tracker walks Quotation → Order confirmed → Production planned → Materials ordered, and the page is built to show the linked production jobs and purchase orders that Odoo spun up automatically from this single confirmation.

4

Production — readiness & the QA gate

Pain · availability + QA

Both manufacturing jobs, with a live material-readiness badge and a QA-gate status. The nested build is visible: the Motor job (WH/MO/00002) is sourced from the mixer job (WH/MO/00001). "Mark done" is blocked until the mandatory quality gates pass — enforced in Odoo, so it can't be skipped from anywhere.

5

Quality Gates — QA that can't be skipped

Pain · skippable QA

Every job carries two mandatory checkpoints — In-Process and Finished-Goods. An inspector records Pass or Fail with a note; a Fail re-opens the gate so the batch must be re-inspected after rework. A manufacturing order simply cannot be closed until both gates pass.

6

Inventory — reorder rules baked in

Pain · no reorder rules

Finished goods and sub-assemblies up top, then all 37 bought-in components with on-hand, reserved, incoming, forecast, and a health badge driven by min/max reorder rules — shortage → replenishment queued or below min → reorder due. Stockouts stop being a surprise.

Straight talk

What's solid, and what's still rough

This is a POC, so here's the honest state — useful before you demo it live.

✓ Working end-to-end

  • ✓Full stack runs: branded UI → JSON-RPC → Odoo → Postgres, on real data
  • ✓Quote → confirm cascades into a mixer MO + nested motor MO automatically
  • ✓Live cost sheet math is correct to the rupee (₹561 motor, ₹1,732.51 mixer)
  • ✓Mandatory QA gates and min/max reorder health are live and enforced
  • ✓Odoo is completely hidden — no leak of URLs, models, or logins

⚠ Known rough edges

  • 1Order page under-reports links. The jobs & POs exist (see Dashboard/Production) but the order-detail panel shows none — a linkage query to fix. Top priority before a live demo.
  • 2Everything shows 0 on-hand / "not available." Expected — we never received the purchase orders yet. Receiving one PO turns readiness green.
  • 3Order total ₹20,616 includes Odoo's default tax on top of the ₹17,325 ex-tax subtotal; tax config is a later step.
  • 4Demo auth only — one hard-coded login; real identity provider is Phase 5.
  • 5Runs locally. The frontend isn't deployed/containerized yet.
The honest headline

The hard part works: a real order flows all the way through manufacturing, costing, QA, and stock, behind a clean white-label UI. The rough edges are finish-work, not foundations — exactly what a POC is meant to prove.
